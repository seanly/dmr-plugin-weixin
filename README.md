# dmr-plugin-weixin

External [DMR](https://github.com/seanly/dmr) plugin (HashiCorp **go-plugin** + `net/rpc`) that connects **Weixin over an ilink/bot HTTP gateway** to the DMR agent: long-poll `getupdates` → `RunAgent` → `sendmessage` (with **`context_token`**). Implements **`ProvideApprover`** for `weixin:p2p:*` tapes (y/s/a/n, same idea as Feishu).

### Design (reference TS → Go → DMR agent)

This is exactly the split you described:

1. **Reference implementation** — [@tencent-weixin/openclaw-weixin](https://www.npmjs.com/package/@tencent-weixin/openclaw-weixin) TypeScript is used **only as a spec**: HTTP paths (`ilink/bot/*`), headers (`AuthorizationType`, per-request `X-WECHAT-UIN`, optional `SKRouteTag`), `base_info`, `get_updates_buf`, `context_token` on sends, and outbound `sendmessage` text shapes. (Media upload / `getuploadurl` / CDN is **not** wired in this build — see below.) The npm package is **not** linked or executed by DMR.

2. **Go implementation** — `gateway_client.go`, `types.go`, `monitor.go`, `reply.go`, … as a **standalone binary** suitable for `go-plugin`.

3. **DMR AI agent integration** — Same external-plugin pattern as [dmr-plugin-feishu](https://github.com/seanly/dmr-plugin-feishu): `SetHostClient` + `Plugin.RunAgent`, tape names `weixin:p2p:<peer>`, dedup / `allow_from`, **`ProvideTools` with `weixinSendText` only**, and `ProvideApprover` for OPA `require_approval`. OpenClaw-specific pieces (plugin-sdk routing, slash commands, pairing files) are **not** copied.

#### Media / file send (temporarily removed)

**`weixinSendFile`** and the **`getuploadurl` → CDN → file/image/video `sendmessage`** path are **not** in this revision (re-add with OpenClaw-style PKCS7 + AES-128-ECB when needed). `cdn_base_url` in config is **optional** until then; `dmr-weixin-login` still writes it for convenience.

**You only need DMR to run it** — settings live under **`plugins[].config`**. **`gateway_base_url`** and **`token`** are required; **`cdn_base_url`** optional for text-only.

## Requirements

- Go **1.25+** (matches DMR)
- `replace github.com/seanly/dmr => ../dmr` (or your checkout path)
- A running **ilink/bot-compatible** gateway and **bot token** (CDN base optional while only text outbound is supported)
- **Not** the WeChat Official Account Open Platform API — this is the personal-channel style `ilink/bot` stack

### Compliance note

Personal Weixin automation may conflict with Tencent’s terms or risk controls. Deploy only if you accept that risk; this README is not legal advice.

## What you put in DMR config (no defaults for URL/token)

These are **not** generated by DMR and have **no universal default**. They must match **your gateway deployment**:

| Key | Meaning |
|-----|--------|
| `gateway_base_url` | Root URL of the HTTP API; the plugin appends paths such as `ilink/bot/getupdates`, `ilink/bot/sendmessage`. Trailing `/` optional. |
| `cdn_base_url` | **Optional** in this build (reserved for future file/image send). If set, use Tencent-style **`https://novac2c.cdn.weixin.qq.com/c2c`** so a future upload path is `…/c2c/upload?…`. |
| `token` | Secret the gateway expects on `Authorization: Bearer <token>`. Put **only the token** in YAML (do **not** include the word `Bearer`). |

**CLI login (optional)** — This repo ships **`dmr-weixin-login`**, which talks to Tencent’s public ilink endpoints (`get_bot_qrcode` → long-poll `get_qrcode_status`), then writes a JSON file with `gateway_base_url`, `cdn_base_url`, and `token`. Point the plugin at that file with **`credentials_path`** (paths relative to your main config file’s directory resolve the same way as `extra_prompt_file`). YAML values for those three keys are **overridden** by non-empty fields in the credentials file. You can still set them only in YAML if you prefer.

The printed **liteapp** URL (`liteapp.weixin.qq.com/...`) often shows only a “open in WeChat” hint in desktop browsers — **no visible QR**. `dmr-weixin-login` also draws the **same URL as a QR in the terminal** (Unicode blocks); scan that with WeChat. Use `-no-terminal-qr` if you only want the link. A small local **web** UI would be the same idea (render a QR for that URL); not shipped here.

How you obtain the three strings without the helper is **outside DMR**: gateway operator docs, `docker-compose`/Helm, or another tool that stored `baseUrl` / `cdnBaseUrl` / `token` for the same protocol.

| Key | Default in plugin |
|-----|-------------------|
| `account_id` | If omitted or empty, the plugin uses **`default`**. Use different values only if you run **multiple** bot identities and need separate `get_updates_buf` files. |

## Build

```bash
make build        # go mod tidy + go build -o dmr-plugin-weixin .
make login-build  # go build -o dmr-weixin-login ./cmd/dmr-weixin-login
make test         # go test ./...
make install      # copies plugin + login helper to ~/.dmr/plugins/
```

**Login once** (writes `~/.dmr/weixin/credentials.json` by default):

```bash
./dmr-weixin-login
# or: go run ./cmd/dmr-weixin-login -- -wait 15m
```

## DMR configuration (`~/.dmr/config.yaml`)

Use **`cli_approver: false`** (or disable any other approver) if this plugin is your only **`ProvideApprover`**.

```yaml
plugins:
  - name: opa_policy
    enabled: true

  - name: cli_approver
    enabled: false

  - name: weixin
    enabled: true
    path: ~/.dmr/plugins/dmr-plugin-weixin
    config:
      # Option A: paste secrets from your gateway
      # gateway_base_url: "https://api.example.com/"
      # cdn_base_url: "https://cdn.example.com"
      # token: "your-bot-token"
      # Option B: after ./dmr-weixin-login — file overlays the three keys
      credentials_path: ~/.dmr/weixin/credentials.json
      # account_id: "default"   # optional; default is default
      allow_from: []
      approval_timeout_sec: 300
      dedup_ttl_minutes: 10
      # extra_prompt / extra_prompt_file — same idea as Feishu plugin
      # sk_route_tag: ""        # optional HTTP header
      # dmr_restart_trigger / dmr_restart_token — same as Feishu (needs allow_from)
```

After merge, **`gateway_base_url`** and **`token`** must be non-empty (`Init` fails otherwise). **`cdn_base_url`** may be empty until file/image outbound is implemented again.

`X-WECHAT-UIN` is generated **per HTTP request** by the plugin; you do not configure it.

Sync cursor `get_updates_buf` is stored under **`config_base_dir/.dmr-weixin/`** (DMR injects `config_base_dir` from the directory of your main config file). If that is missing, the OS user cache dir is used.

## Tape names

Inbound private chat: **`weixin:p2p:<from_user_id>`** (e.g. `xxx@im.wechat`). One global worker processes jobs in order (same pattern as Feishu).

## Tools

- **`weixinSendText`** — Plain text to current peer when run from inbound context; for cron use `tape_name` or `peer_id`. Needs a **cached `context_token`**. Set **`DMR_WEIXIN_DEBUG_SEND=1`** on `dmr serve` to log truncated `sendmessage` JSON when debugging delivery.

## Approvals

Only **`weixin:p2p:*`** tapes. Plain-text prompts. Replies: **y / s / a / n** (single), or batch **yes / 1,3,5** style, aligned with CLI approver semantics.

## Cron

Use tape **`weixin:p2p:<peer_id>`** with `dmr-plugin-cron`; deliver with **`weixinSendText`** and `tape_name` / `peer_id` — same pattern as Feishu + `feishuSendText`.
